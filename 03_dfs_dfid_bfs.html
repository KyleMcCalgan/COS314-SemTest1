DFS · DFID · BFS

The three fundamental uninformed search strategies

Depth First Search (DFS)

DFS dives as deep as possible down one branch before backtracking. It uses a stack (LIFO) — the OPEN list in the algorithm always has new children placed at the left end (front).

The Core Idea Go deep first. When you hit a dead end or exhaust a branch, backtrack and try the next option. Like exploring a maze by always turning left.
def dfs(in Start, out State) open = [Start] // stack (LIFO) closed = [] State = failure while (open ≠ []) AND (State ≠ success) X ← remove LEFTMOST node from open if X == goal State = success else children ← generate children of X put X on closed eliminate children already on open or closed // avoid revisits put remaining children on LEFT end of open // ← KEY: left = DFS return State
The ONLY difference between DFS and BFS in code is WHERE children are added: DFS = LEFT end (stack). BFS = RIGHT end (queue). Memorise this!

Worked Example

Legal moves: A→B,C | B→D,E | C→F,G | D→I,J | I→K,L    Start: A, Goal: J

Step | OPEN | CLOSED | X | X's Children ─────┼───────────────┼────────────┼────┼───────────── 1 | [A] | [] | | 2 | [] | | A | B,C 3 | [B,C] | [A] | | 4 | [C] | | B | D,E 5 | [D,E,C] | [A,B] | | 6 | [E,C] | | D | I,J 7 | [I,J,E,C] | [A,B,D] | | 8 | [J,E,C] | | I | K,L 9 | [K,L,J,E,C] | [A,B,D,I] | | 10 | [L,J,E,C] | | K | none 11 | [J,E,C] | [A,B,D,I,K]| | 12 | [E,C] | | J | → SUCCESS ✓

Order visited: A → B → D → I → K → J ✓

Properties of DFS

Time
O(b^m)

b=branching, m=max depth

Space
O(bm)

Only one path in memory

Optimal?
✗ No

May find long suboptimal path

  • Can get "lost" — may follow infinite branches (use depth bound)
  • More efficient when branching factor is high
  • Solution path may be long and suboptimal
  • Low memory usage — only stores one path at a time

Depth First Search with Iterative Deepening (DFID)

DFID solves DFS's main problem (getting lost in deep branches) by running DFS repeatedly with increasing depth limits. Each iteration allows one more level of depth.

The Core Idea Run DFS with depth limit = 1. If not found, increase to 2. Keep increasing until found. This gives DFS the space efficiency + BFS's completeness (always finds shortest path).
procedure DFID(initial_state, goal_states) search_depth = 1 while (solution path not found) run dfs(initial_state, goal_states) with depth bound = search_depth search_depth = search_depth + 1

Example — Iterating through depth levels

Iteration 1 (depth limit = 1): Open: [A] → expand A → [B,C] B hits limit, C hits limit → FAIL Iteration 2 (depth limit = 2): A → B → D (limit), E (limit) ← D and E hit limit A → C → F (limit), G (limit) ← fail Iteration 3 (depth limit = 3): A → B → D → I, J → J found! → SUCCESS ✓
DFID seems wasteful (re-visits earlier nodes), but nodes at early depths are exponentially fewer than deep nodes. The overhead is small. DFID is GUARANTEED to find the shortest path.

When to prefer DFID over plain DFS

  • State space has infinite depth (DFS would loop forever)
  • You need guarantee of shortest path (like BFS) but with less memory
  • You don't know the depth of the solution in advance
Space
O(bd)

Like DFS — very efficient

Optimal?
✓ Yes

Finds shortest path

Complete?
✓ Yes

Always terminates

Breadth First Search (BFS)

BFS explores all nodes at one level before moving to the next. It uses a queue (FIFO) — new children are added to the right end of OPEN.

The Core Idea Explore level by level, left to right. Never go deeper until all nodes at the current depth are visited. Guarantees the shortest path is found first.
def bfs(in Start, out State) open = [Start] // queue (FIFO) closed = [] State = failure while (open ≠ []) AND (State ≠ success) X ← remove LEFTMOST node from open if X == goal State = success else children ← generate children of X put X on closed eliminate children already on open or closed put remaining children on RIGHT end of open // ← KEY: right = BFS return State

Worked Example (same problem as DFS)

Step | OPEN | CLOSED | X | X's Children ─────┼─────────────────┼──────────────┼────┼───────────── 1 | [A] | [] | | 2 | [] | | A | B,C 3 | [B,C] | [A] | | 4 | [C] | | B | D,E 5 | [C,D,E] | [A,B] | | ← D,E added to RIGHT 6 | [D,E] | | C | F,G 7 | [D,E,F,G] | [A,B,C] | | 8 | [E,F,G] | | D | I,J 9 | [E,F,G,I,J] | [A,B,C,D] | | 10 | [F,G,I,J] | | E | (goal!) → SUCCESS ✓

BFS visits nodes level by level: A | B,C | D,E,F,G | I,J,... — always finds the SHALLOWEST goal first.

Properties of BFS

Time
O(b^d)

d=depth of solution

Space
O(b^d)

Stores entire frontier — BAD

Optimal?
✓ Yes

Always finds shortest

  • Guaranteed to find the shortest path (fewest steps)
  • High branching factor causes combinatorial explosion — huge memory use
  • Can be slow when solution is deep

DFS vs BFS vs DFID — Side by Side

Property BFS DFS DFID
Data structure Queue (FIFO) Stack (LIFO) Stack (LIFO)
Add children to RIGHT of open LEFT of open LEFT of open
Optimal (shortest path)? ✅ Yes ❌ No ✅ Yes
Complete? ✅ Yes ❌ No (needs depth bound) ✅ Yes
Memory use HIGH O(b^d) LOW O(bm) LOW O(bd)
Time use O(b^d) O(b^m) O(b^d)
Best when Short solution, small branching factor High branching factor, deep search OK Unknown depth, need shortest path + low memory
For the test: BFS = RIGHT end. DFS = LEFT end. DFID = DFS + increasing depth limit. DFID gives you the best of both worlds: BFS optimality + DFS memory efficiency.
