Metaheuristics that escape local optima

Simulated Annealing (SA)

Simulated annealing is like hill climbing that sometimes accepts worse solutions. This counterintuitive approach lets it escape local optima. Named after metal annealing: heat metal high, cool slowly → stable low-energy crystal structure.

The Core Insight Hill climbing only moves "uphill" (better solutions). This gets stuck at local optima. SA occasionally moves "downhill" (worse solutions), allowing it to escape and find the global optimum.

Temperature and Probability

The key parameter is temperature T. It controls how likely SA is to accept a worse solution:

  • High T (hot): More likely to accept worse solutions — explore widely
  • Low T (cold): Rarely accept worse solutions — exploit what's been found
  • Acceptance probability for worse solution: P = min(1, exp(−ΔL/T))
Hot (explore)
Cold (exploit)
SIMULATED ANNEALING: Input: Initial state s, loss function L(·) CURRENT ← s BEST ← s T ← T₀ (initial temperature — similar to average loss difference) t ← 1 repeat: NEXT ← randomly chosen adjacent state to CURRENT ΔL ← L(NEXT) − L(CURRENT) // negative = NEXT is better if ΔL < 0: // NEXT is better → always accept CURRENT ← NEXT else: // NEXT is worse → accept with probability CURRENT ← NEXT with probability min(1, exp(−ΔL/T)) if L(CURRENT) < L(BEST): BEST ← CURRENT t ← t + 1 T ← T₀ / log(t + 1) // cooling schedule until no improvement in BEST for N iterations return BEST
ΔL = L(next) − L(current). If ΔL < 0, next is BETTER (lower loss) → always accept. If ΔL > 0, next is WORSE → accept with probability exp(−ΔL/T). As T decreases, this probability shrinks.

Cooling Schedule

The cooling schedule determines how temperature decreases. Common example: T = T₀ / log(t+1). The exact schedule is a key parameter — too fast = local optima, too slow = computationally expensive.

Disadvantages of SA

Limitations
  • No guarantee of finding global optimum — heuristic
  • Heavy parameter tuning needed (T₀, cooling schedule, neighbourhood)
  • Can be slow to converge
  • Computationally expensive for complex problems

Tabu Search

Tabu search is like hill climbing that keeps a "tabu list" of recently visited states — a short-term memory that prevents revisiting recent solutions, forcing exploration of new areas.

The Core Idea Unlike SA (probabilistic), Tabu Search is deterministic. It remembers where it's been (tabu list) and forbids going back there for a while. This creates a structured way to escape local optima.
TABU SEARCH: s ← s₀ // initial candidate solution L ← {} // tabu list (empty) z ← max tabu length // FIFO queue bound repeat: s' ← randomly generate a neighbour of s if s' ∉ L: // not forbidden if length(L) > z: remove OLDEST solution from L // FIFO — evict old entries add s' to L if s' is better than s: s ← s' until stopping criterion is met return s

How the Tabu List Works

  • The tabu list stores recently visited solutions (or moves)
  • Has a fixed maximum size z (FIFO — oldest entry removed first)
  • Solutions on the tabu list cannot be revisited (they are "taboo")
  • Forces the algorithm to explore new regions of the search space
  • Comparable to the CLOSED list in standard search algorithms
The tabu list uses FIFO (First-In First-Out). When it's full, the OLDEST entry is removed. This gives the algorithm "short-term memory" — recently visited states are forbidden, but old ones eventually become available again.

Tabu vs Hill Climbing

Hill Climbing
  • Always accepts better moves
  • No memory
  • Gets stuck at local optima permanently
  • Simple
Tabu Search
  • Can accept worse moves (to escape)
  • Short-term memory (tabu list)
  • Escapes local optima via forbidden zones
  • Usually outperforms hill climbing

Two Practical Enhancements

Enhancement 1: Best-neighbour selection Instead of any random unvisited neighbour, pick the BEST unvisited neighbour according to the cost function. Slower per step, but faster improvement.
Enhancement 2: Bounded tabu list Keep the list at a fixed max size. When full, remove the earliest entry. This is important for computational and memory efficiency.

Disadvantages of Tabu Search

Limitations
  • No guarantee of finding global optimum
  • Performance depends heavily on tabu list size and parameters
  • Stopping criteria are tricky — relies on randomness so may run indefinitely
  • Memory-constrained environments limit tabu list size
  • Initial solution affects where the search begins

Iterated Local Search (ILS)

ILS improves upon basic local search by repeatedly perturbing locally optimal solutions and applying local search again. It escapes local optima by making a big jump to a new area, then fine-tuning from there.

The Core Idea Local search finds a local optimum. Then PERTURB it (big random jump). Apply local search again from there. Accept the new result if it's better. Repeat. This explores multiple regions.
ITERATED LOCAL SEARCH (ILS): s₁ ← GenerateInitialSolution() s* ← LocalSearch(s₁) // find first local optimum repeat: s' ← Perturbation(s*, history) // jump to new area s*' ← LocalSearch(s') // find local optimum from there s* ← AcceptanceCriterion(s*, s*', history) // keep better one until termination condition met return s*

Step-by-Step Breakdown

  • 1. Generate Initial Solution: Random or heuristic-guided starting point
  • 2. Local Search: Find a local optimum from the starting point
  • 3. Perturbation: Make a BIG change (bigger than local search moves) to escape the local optimum basin. Strength is a key parameter.
  • 4. Local Search again: Fine-tune the perturbed solution to a new local optimum
  • 5. Acceptance Criterion: Decide whether the new local optimum is better. May accept worse solutions occasionally (like SA).
  • 6. Repeat until stopping condition (iterations/time/target quality)
KEY DIFFERENCE from SA/Tabu: ILS makes LARGE perturbations to jump across basins of attraction, then uses a LOCAL search to find the best solution in that new region. It's a macro-level diversification strategy.
